# 4주차: Zero-Touch 1인 개발 환경 고도화 — 작업 완료 보고

> **프로젝트:** AI 기반 2D 횡스크롤 액션 게임  
> **개발자:** 장석원  
> **환경:** Unity + Claude 4.6 + Unity MCP (Stdio 전송)  
> **에셋 도구:** 나노 바나나 (Gemini 3 Flash Image)  
> **작업 기간:** 4주차

---

## 1. 에이전트 기반 에셋 적용 — 핵심 작업

### 1-1. 작업 원칙: 드래그 앤 드롭 금지

4주차의 핵심 목표는 나노 바나나로 생성한 에셋을 Unity 프로젝트에 적용하되, 개발자가 직접 드래그 앤 드롭으로 배치하는 것이 아니라 Unity MCP를 통해 에이전트가 직접 에셋을 적용하도록 하는 것이었다. 사람이 하는 일은 나노 바나나에서 이미지를 생성하고 누끼를 따는 것까지이며, 이후 Unity 내 배치와 설정은 전부 MCP 명령으로 수행했다.

### 1-2. 첫 번째 시도: 타일맵 방식 — 실패

처음에는 타일맵(Tilemap) 방식으로 배경과 환경을 구성하려 했다. 나노 바나나로 타일 단위 이미지를 생성한 뒤 MCP를 통해 타일맵에 배치하는 방식이었다. 그러나 타일맵은 개별 타일 하나하나를 좌표 지정하여 배치해야 하기 때문에 MCP 호출 횟수가 급증했고, 토큰이 금방 소진되어 작업 지속이 불가능했다. 타일맵 하나를 채우는 데만 수십 회의 MCP 호출이 필요했고, 응답 지연까지 겹치면서 실질적인 작업 효율이 수동 배치보다 오히려 낮았다.

### 1-3. 두 번째 시도: 오브젝트 단위 큼직한 이미지 — 결과물 불만족

타일맵의 토큰 소진 문제를 우회하기 위해, 오브젝트 단위로 큼직하게 이미지를 뽑아서 통째로 적용하는 방식으로 전환했다. 배경을 타일 조각이 아닌 하나의 넓은 이미지로 생성하고, 환경 오브젝트도 개별 스프라이트로 처리하여 MCP 호출 횟수를 대폭 줄였다. 토큰 문제는 해결되었지만, 결과물의 비주얼 퀄리티가 기대에 미치지 못했다. 큼직한 이미지 한 장으로 배경 전체를 표현하다 보니 디테일이 부족했고, 오브젝트 간 아트 스타일의 일관성도 떨어졌다.

### 1-4. 멀티 에이전트 오케스트레이션 활용

비주얼 퀄리티를 끌어올리기 위해 2주차에 구축한 멀티 에이전트 오케스트레이션 파이프라인을 실전 투입했다. 워크플로우는 다음과 같았다.

먼저 Claude에게 기획자 역할을 부여하고, 게임의 전체 아트 방향성과 각 씬에 필요한 에셋 목록을 정리하도록 했다. 기획자 에이전트가 정리한 요구사항을 아트 디렉터 역할의 에이전트에게 전달하여 나노 바나나용 프롬프트를 생성하도록 지시했다. 이렇게 뽑아낸 프롬프트를 내가 직접 나노 바나나에 입력하여 에셋을 생성했고, 생성된 에셋을 누끼 딴 뒤 MCP에게 Unity 프로젝트 내 적용을 명령했다.

```
[기획자 에이전트] 씬별 필요 에셋 목록 + 아트 방향성 정리
       ↓
[아트 디렉터 에이전트] 나노 바나나용 프롬프트 생성
       ↓
[개발자(나)] 나노 바나나에 프롬프트 입력 → 이미지 생성 → 누끼 작업
       ↓
[Unity MCP] 에셋 임포트 + Sprite 설정 + 씬 배치
```

오케스트레이션을 적용한 뒤 프롬프트 품질이 올라가면서 에셋의 스타일 일관성은 이전보다 나아졌다. 그러나 솔직히 결과물이 만족스러운 수준은 아니었다. 나노 바나나가 생성하는 이미지 자체의 해상도와 게임 에셋으로서의 적합성에 한계가 있었고, 누끼 작업 후 Unity에 올리면 엣지가 거칠거나 배경 잔상이 남는 문제가 반복되었다.

### 1-5. 그럼에도 여러 방법을 계속 시도

결과물이 마음에 들지 않았지만 아랑곳하지 않고 다양한 접근을 계속 시도했다.

프롬프트를 세분화하여 같은 오브젝트를 여러 스타일로 생성한 뒤 가장 나은 것을 선택하는 방식을 시도했다. 배경을 레이어별로 분리 생성하여(먼 배경, 중간 배경, 전경) 패럴랙스 효과와 함께 합성하는 방식도 적용했다. 나노 바나나의 출력 이미지를 후처리하여 색감 보정이나 해상도 업스케일링을 거친 뒤 Unity에 적용하는 시도도 했다. MCP를 통해 SpriteRenderer의 Color 속성을 조정하거나 Material의 Tint를 변경하여 Unity 쪽에서 비주얼을 보정하는 방법도 병행했다.

이러한 시행착오의 과정 자체가 "AI 에셋 생성 도구의 현실적 한계"와 "MCP를 통한 자동화의 실제 가능 범위"를 체감하는 귀중한 데이터가 되었다.

## 2. 시간 측정 기록

에이전트를 모든 작업에 적용하면서 각 단계별 소요 시간을 측정했다.

### 2-1. 에셋 적용 작업 시간

| 작업 | 소요 시간 | 비고 |
|------|----------|------|
| 타일맵 방식 시도 → 실패 판단까지 | 1시간 | 토큰 소진으로 중단, 방식 자체를 폐기 |
| 오브젝트 단위 전환 후 에셋 적용 | 4시간 | 여러 에셋을 큼직하게 뽑아 적용하는 전 과정 포함 |
| 오케스트레이션 1사이클 (프롬프트 생성 → 나노바나나 → 누끼 → MCP 적용) | 약 5분/회 | 파이프라인 안정화 후 기준 |
| MCP 연결 끊김 복구 | 약 1분/회 | 축적된 해결 절차 적용 (이전 주차 대비 대폭 단축) |

타일맵 방식에 1시간을 투자한 뒤 토큰 소진으로 폐기한 것은 시간 낭비처럼 보이지만, 이 실패를 통해 "MCP 자동화에서 호출 횟수를 최소화하는 설계"라는 핵심 원칙을 얻었다. 오브젝트 단위로 전환한 뒤에는 4시간 만에 게임 전체의 주요 에셋을 적용할 수 있었다.

특히 오케스트레이션 파이프라인이 안정화된 이후에는 1사이클(기획자가 프롬프트 뽑기 → 나노바나나 이미지 생성 → 누끼 → MCP로 Unity 적용)이 약 5분 만에 완료되었다. 수동으로 같은 작업을 할 경우 에셋 검색, 다운로드, 임포트 설정, 드래그 앤 드롭 배치, 컴포넌트 설정까지 최소 15~20분은 걸리는 것을 감안하면, 반복 작업에서의 시간 절감 효과가 컸다.

### 2-2. MCP 복구 시간의 변화

MCP 연결 끊김 복구 시간은 주차별로 뚜렷한 개선을 보였다. 3주차에는 "No Unity Editor instances found" 에러가 발생했을 때 원인 파악부터 해결까지 30분 이상 걸렸으나, 해결 절차(Unity MCP 패널 세션 종료 → Regenerate → Claude Desktop 완전 재시작)를 지식 베이스로 정리한 이후 4주차에는 동일 이슈가 발생해도 약 1분 만에 복구할 수 있었다. 트러블슈팅 지식을 축적하는 것만으로도 작업 효율이 크게 올라간다는 것을 수치로 확인한 사례다.

## 3. 트러블 슈팅 기록

### 3-1. 토큰 소진 문제 (타일맵 시도)

타일맵 방식은 하나의 맵을 구성하는 데 타일 개수 × MCP 호출이 필요했기 때문에 토큰이 급격히 소모되었다. 30×20 크기의 타일맵만 해도 수백 회의 배치 명령이 필요했고, 한 세션의 토큰 한도 내에서 완성이 불가능했다. 이를 계기로 "MCP 자동화의 효율은 호출 횟수에 반비례한다"는 교훈을 얻었으며, 이후 모든 작업을 가능한 한 적은 호출로 달성하는 방향으로 설계를 전환했다.

### 3-2. MCP 세션 안정성

이전 주차에서 겪었던 "No Unity Editor instances found" 에러가 이번 주에도 간헐적으로 발생했다. 이미 축적한 해결 절차(Unity MCP 패널 세션 종료 → Regenerate → Claude Desktop 완전 재시작)를 적용하여 매번 5분 이내에 복구했다. 3주차에는 같은 이슈에 30분 이상 소모했던 것과 비교하면, 트러블슈팅 지식 축적의 효과가 분명하게 나타났다.

## 4. 핵심 인사이트

### 4-1. AI 에셋 생성의 현실

나노 바나나를 통한 게임 에셋 생성은 "빠르게 프로토타입 수준의 비주얼을 입히는 것"에는 유효하지만, 상용 수준의 아트 퀄리티를 기대하기는 어려웠다. 특히 게임 에셋은 단순 이미지와 달리 타일링 적합성, 해상도 일관성, 투명도 처리 등 기술적 요구사항이 있어서, 생성 AI의 출력물을 그대로 쓰기보다 후처리 파이프라인이 필수적이라는 것을 체감했다.

### 4-2. 토큰 효율 설계의 중요성

타일맵 실패 경험에서 배운 가장 큰 교훈은, MCP 기반 자동화를 설계할 때 "호출 횟수 대비 작업량"을 사전에 산출해야 한다는 것이다. 같은 결과를 만들더라도 타일 단위 배치와 오브젝트 단위 배치는 토큰 소모량이 수십 배 차이가 났다. Zero-Touch를 지향하되, 토큰 예산 내에서 실현 가능한 단위를 설계하는 것이 실무적으로 중요했다.

### 4-3. 오케스트레이션의 실전 효과

멀티 에이전트 오케스트레이션은 "프롬프트 품질을 체계적으로 높이는 도구"로서 효과가 있었다. 기획자 → 아트 디렉터의 파이프라인을 거치면 나노 바나나에 입력하는 프롬프트의 구체성과 일관성이 확실히 올라갔다. 결과물의 절대적 퀄리티는 여전히 한계가 있었지만, 프롬프트를 직접 즉흥으로 작성할 때보다 시행착오 횟수가 줄어드는 효과는 분명했다.

### 4-4. "별로여도 계속 시도하는 것"의 가치

결과물이 만족스럽지 않아도 여러 방법을 계속 시도한 것은 발표 자료 측면에서 오히려 풍부한 데이터가 되었다. 타일맵 실패 → 오브젝트 단위 전환 → 오케스트레이션 적용 → 후처리 병행이라는 시행착오의 흐름 자체가 "AI 도구의 실전 적용에서 실제로 부딪히는 문제들"을 보여주는 사례가 된다. 성공 사례만 나열하는 것보다 이런 시행착오 과정이 더 설득력 있는 발표 콘텐츠가 될 수 있다.
