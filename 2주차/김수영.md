## 초안

또 하나 중요한 차이 — 별도 서브에이전트가 아닙니다. /plan 을 호출해도 새 컨텍스트의 분리된 에이전트가 뜨는 게 아니라, 현재 메인 대화의 Claude가 그 프롬프트를 받고 동작 모드를 바꾸는 것입니다. 진짜 컨텍스트 격리가 필요하면 Agent툴(서브에이전트, .claude/agents/*.md)을 써야 합니다.
이 분리 방식의 장단점

  장점
  - 역할 명확화 — 한 호출에 한 책임. "기획 도중에 코드부터 짜버리는" 흐름을 막아 줌
  - 컨벤션 강제 — /dev 프롬프트에 o/_ 프리픽스, 서버 권위, 데이터드리븐 규칙이 박혀 있어 매번 상기시킬 필요 없음
  - 체크리스트 효과 — /qa 가 리플리케이션·GAS·라이프사이클을 체계적으로 훑어줌. 사람이 매번 떠올리기 어려움
  - 출력 포맷 일관성 — Plan/QA가 표·섹션으로 고정 → 다음 단계로 넘기기 쉬움
  - 가볍고 즉시 수정 가능 — 그냥 .md 파일이라 git 추적·팀 공유 용이

  단점
  - 컨텍스트 공유 안 됨 — 서브에이전트가 아니라서, /plan 출력을 /dev 에 다시 붙여넣어야 함. 한 세션에서 자연스레 이어지지 않음
  - 수동 워크플로우 강제 — 사용자가 "plan → dev → qa" 순서를 직접 지켜야 함. 잊으면 의미 없음
  - CLAUDE.md 와 중복 — 컨벤션이 이미 CLAUDE.md에 있어서 일부는 이중 관리. 한쪽 바뀌면 다른 쪽도 갱신해야 함
  - 유연성 손실 — /plan 은 "코드 안 씀"이 강제라, 작은 1줄 수정도 계획 단계 거치면 과한 오버헤드
  - 서브에이전트가 아니라 진짜 격리·병렬 실행이 안 됨 — 정말 큰 작업이면 .claude/agents/ 로 옮기는 게 나을 수 있음

  추천 사용 기준: 새 기능·리팩터링처럼 사전 설계가 필요한 작업에는 /plan → /dev → /qa 풀코스. 한 줄 버그 수정·텍스트 변경은 그냥 메인 대화로 처리하는 게 효율적입니다.
