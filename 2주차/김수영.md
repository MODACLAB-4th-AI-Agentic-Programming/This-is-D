

**이번주 주제** : 에이전트를 나누고 역할 부여 및 사용

### **사전 지식**

멀티 에이전트 오케스트레이션이란?

**단일 에이전트의 한계**

- 컨텍스트 윈도우 제한
- 하나의 작업에 모든 역할을 다 담으면 프롬프트가 비대해지고 품질 저하
- 병렬 처리 불가

**오케스트레이션의 접근**

- 역할별로 전문화된 에이전트를 분리 (예: Planner / Coder / Reviewer / QA)
- 상위 "오케스트레이터"가 작업을 분해하고 적절한 에이전트에 위임
- 각 에이전트는 독립된 컨텍스트에서 자기 역할만 수행 후 결과 반환

<aside>
💡

기존에 세션을 통해 컨텍스트를 분리하는 것을, 하나의 세션 내에서 다른 에이전트를 쓰는 것처럼 사용할 수 있는 것이구나!

</aside>

### **실습 과정**

1. 에이전트 분리 요청

![image.png](attachment:8635a66b-538f-435d-8c62-49e13d83c921:image.png)

**각 커맨드 역할**

| 커맨드 | 역할 |
| --- | --- |
| **/plan** | 요구사항 분석 → 영향 파일·구현 단계·GAS 고려사항·리스크를 구조화된 계획으로 출력 |
| **/dev** | 프로젝트 컨벤션(`o` 프리픽스, `_` 언더스코어, 서버 권위, 데이터드리븐 패턴) 준수하며 구현 |
| **/qa** | 컴파일·네이밍·리플리케이션·GAS·라이프사이클·하드코딩 이슈를 표 형식으로 리뷰. `fix` 키워드 시 수정까지 |

**권장 사용 흐름**

`/plan <요구사항>` → 계획 검토 → `/dev <플랜 내용>` → `/qa` 로 검증

**분리한 이유**

단일 대화에서 기획·구현·검증을 한꺼번에 처리하니 컨텍스트가 섞이고 컨벤션 준수가 흔들려서, 역할별 시스템 프롬프트를 분리해 책임을 명확히 함

---

**특이한 점 / 좋았던 점 / 나빴던 점**

- 장점이라고 생각함
    - 채팅 목록에서도 컨텍스트 스위칭 할 수 있지만
    - 한 세션 안에서 흐름이 이어짐
    - 한 주제 안에서 기획, 개발, QA를 나눠서 할 수 있어짐.
